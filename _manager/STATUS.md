- [x] Той самий voice picker застосований і в Text Mode (`speakCurrentBlock`), а не лише в Words Mode — раніше там взагалі не було явного вибору голосу

## BUG-FIX (регресія): вірш "вивчається без перших рядків" (2026-07-21, продовження)

**Скарга User:** "Обираєш 5хв часу є і потім починає вірш вивчення без перших двох. Чому так?"

**Корінь (знайдено CEO, не менеджером-агентом):** це РЕГРЕСІЯ від сьогоднішнього ж фіксу #3 зі списку 16 ("smartSplitText ігнорує blockSize"). У `splitLongSentence()` (app.js, викликається коли рядок вірша/речення довший за blockSize) фінальний фільтр `else if (countWords(piece) >= 4) { final.push(piece); }` МОВЧКИ ВІДКИДАВ будь-який шматок коротший за 4 слова замість того, щоб об'єднати його з сусіднім (як стверджував коментар над кодом, але код цього не робив). Типовий рядок вірша виду "Ранок настав, і сонце сяє" після розбиття по комі дає шматки "Ранок настав," (2 слова) і "і сонце сяє" (3 слова) — обидва <4 слів → ОБИДВА зникали з тексту повністю. Не пов'язано з вибором "5 хвилин" напряму — просто в тексту User перші рядки вірша, ймовірно, довші за обраний розмір блоку і містять кому, тому саме на них спрацював баг найпомітніше.

**Підтверджено відтворенням:** вірш з 4 рядків, кожен виду "Х, і Y" — `smartSplitText` губив по одному шматку з КОЖНОГО рядка (не тільки перших двох), 8 з ~12 очікуваних блоків замість повного покриття.

**Рішення:** `splitLongSentence()` — короткий шматок (<4 слів) тепер приєднується до ПОПЕРЕДНЬОГО шматка (`final[final.length-1] += ' ' + piece`) замість відкидання; якщо це найперший шматок (нема куди приєднати) — лишається окремим блоком, а не губиться.

Файли змінено: `app.js` (`splitLongSentence`), `index.html` (`?v=` app.js 76→77)
Перевірено: тестовий вірш з коротких коматизованих рядків — сума слів на вході й виході тексту тепер однакова (29=29, раніше губились слова); короткі беспунктуаційні рядки (оригінальний сценарій без коми) — без змін, як і було; повний прогін через реальний UI (paste→setup→5 хв→Start) — перший блок тепер саме перший рядок вірша, не третій; немає console.error

**Root cause #2:** `.btn-check`/`#writeCheckBtn` використовував `var(--primary)`, який у темній темі — світлий індиго `#818cf8`. Разом з білим текстом і додатковою непрозорістю 0.5 (disabled-стан) контраст падав до ~3.3:1 — нижче порогу WCAG AA (4.5:1). Той самий клас проблеми, що вже був знайдений і виправлений для `.btn-wt-check` у Words Mode (21.07) через `#4f46e5` — цей фікс просто ніколи не застосували до Text Mode-кнопки.
